# Dossier: What Changes, What Doesn't, and What Breaks First

Dossier is a product-building environment where a human thinks in outcomes and an AI thinks in structure. You describe what you want to build — or connect a repo of something you've already built — and Dossier turns that conversation into a living map: projects decomposed into workflows, workflows into activities, activities into cards. Each card carries its own requirements, assumptions, known facts, open questions, and planned files. When the map is right, you finalize it, and AI agents write the code on feature branches. You review, iterate, merge. That's the loop.

## What the human can now do that they couldn't before

A single person can hold an entire product in their head without it falling apart. Before Dossier, building with AI meant pasting the same context into twenty chat windows, hoping the model remembered what you said three prompts ago, and manually stitching outputs together. Dossier eliminates that. You describe an idea once, the planning layer structures it, and every downstream agent gets exactly the context it needs — scoped to the card it's working on, not the whole project. You can go from a paragraph-long idea to a buildable, branch-per-card codebase in a session. You can also point it at an existing repo and get a structured map back, then make changes through the same card-based flow. The human stays in the chair that matters: deciding what to build, reviewing what was built, and choosing when to ship.

## What AI is responsible for

AI does two distinct jobs. The planning LLM takes unstructured intent and produces structured output: workflows, activities, cards, knowledge items, planned files, and project-wide context artifacts like architectural summaries and data contracts. It asks clarifying questions when the intent is ambiguous. It never generates production code. The build agents take finalized cards — with their approved planned files, requirements, and scoped context retrieved from a local vector index — and write code on isolated feature branches. They commit, run quality checks, and surface the results. Both roles operate within hard boundaries: the planning LLM cannot trigger builds or write to GitHub; the build agents cannot merge to main or skip checks.

## Where AI must stop

AI doesn't decide what gets built, what's good enough, or what ships. The human approves planned files before finalization. The human triggers finalization and builds — there's no auto-pilot mode. The human reviews produced code and requests changes through chat. The human merges pull requests on GitHub. These aren't soft defaults; they're structural constraints in the system. The planning LLM's output is validated against schemas that explicitly reject code-generation intent. Build agents write to feature branches, never to main. There is no path through the system where AI ships something the human hasn't reviewed.

## What would break first at scale

The local SQLite database and in-process WASM embeddings are the obvious first bottleneck. Dossier currently stores everything — project state, planning actions, memory units, embeddings — in a single SQLite file on disk. That's fine for one person building one product. It won't survive concurrent users, large repos, or teams. The finalize step already parallelizes five LLM calls; at scale, those calls would need queuing, rate-limit handling, and retry logic that doesn't exist yet. The action pipeline — every map mutation flows through a single validate-then-apply chain with idempotency keys — is clean but serial; concurrent edits to the same project would need conflict resolution. And the vector retrieval layer (RuVector running in WASM) would need to move to a proper service with persistent indexes, not in-process memory. The architecture is sound for its current scope. Scaling it means replacing the storage and compute layers underneath without changing the human-AI contract on top.
